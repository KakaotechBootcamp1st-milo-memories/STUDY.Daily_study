# 주니어가 개발자가 쉽게 빠지는 설레발 주도 개발
## "그 기술 좋다더라"
- 소프트웨어 개발 팀들은 자주 소프트웨어 구조 설계나 기술 스택을 결정한다.
- 설레발 주도 개발은 제대로 된 연구나 자신들의 프로젝트

## 설레발 주도 개발자
- 레딧 주도 개발
- 컨퍼런스 주도 개발
- 목소리 큰 놈 주도 결단
- 라이브러리 주도 개발
- 지식인 주도 개발

## 설레발 주도 개발의 단계
1. 진짜 문제와 해결책
- 문제가 생겼지만 현재의 기술 스택, 구조로는 해결이 불가능하다고 판단
- 새로운 프레임워크나 라이브러리를 만들거나 도입으로 해결을 함

2. 발표, 자랑질, 키워드
- 새로운 프레임워크와 라이브러리의 도입으로 그들의 문제가 해결되고 좋은 결과물을 내기 시작함
- 발표, 블로그에 포스트쓰기, 컨퍼런스에 나가서 약을 팔기
- 그들에게는 “해결했다”라는 것이 중요한 문제였기 때문에 이 해결책을 자랑스럽게 뽐내기 시작함

3. 빠돌이 탄생
- 블로그 글, 컨퍼런스의 발표를 듣고 많은 설레발 주도 개발자들이 새로운 해결책을 도입함

4. 실망
- 사람들이 기대했던 것만큼 팀에 도움이 되지 않고 오히려 더 일을 늘리기만 함

5. 현실 인식
- 새로운 기술 도입에 따른 대가가 무엇인지, 무엇을 위한 기술이었는지 깨닫기 시작함

## 예시: MSA 아키텍쳐
1. 커다란 단일 어플리케이션은 확장이 어렵습니다. 이걸 여러 서비스로 나눠버리면 초당 요청 처리수의 관점에서나 여러개의 팀의 협업 관점에서나 확장하기 쉬워집니다.
2. 설레발 키워드: 확장성, 느슨한 의존성, 단일 어플리케이션
3. 뭐든지 다 서비스로 재작성하자! 우리 코드가 ‘스파게티 코드’인 건 다 단일 어플리케이션 구조 때문이야! 모든걸 마이크로 서비스로 다시 만들어야해!
4. 앱 개발이 넘나 느리고 배포가 엿같은데다 여러 시스템에서 버그를 추적하는 게 넘나 빡세…
5. 마이크로서비스는 팀에 아주 많은 개발을 요구하고, 마이크로서비스가 아니었음에도 충분히 해결할 수 있는 문제였음을 깨달음

- 넘쳐나는 아이디어를 그릴 도화지의 부족
- 슬랙, 클릭업, 구글 워크스페이스의 API 리밋
- 시트 과부하
### 2단계: 키워드와 스택 리서치

#### 1.	결정하기 전에 연구하고 테스트하기

# 잘 훔치고 잘 따라하기

## 제품을 애자일하게
1. 스프린트
- 팀 구성원이 특정 목표를 위해 작업하는 고정된 기간
2. 일일 스크럼
- 스크럼마스터가 15분 이내로 매일 같은 시간과 장소에서 서서 어제 한 일, 오늘 할 일을 팀원들과 공유
- 스프린트 목표에 대한 진행 상황과 목표를 방해할 수 있는 문제를 공유
3. 스프린트 검토 및 회고
- 완료한 작업에 대해 이해 관계자와 공유하고 피드백
- 앞으로 다가올 계획에 대해 협의
- 이번 스프린트의 강점과 약점, 향후 개선 영역, 지속적인 프로세스 개선 작업을 위해 분석하는 회의
4. 백로그 정제
- 다음 스프린트를 위해 백로그를 수정하고 우선순위를 정하는 프로세스

## 팀을 애자일하게
1. KPT 회고 도입
- Keep, Problem, Try
2. 위임하기
- 업무 위임
3. 플래닝 포커 도입
- 업무의 공수를 계산해서 업무를 분배함

### 플래닝 포커
- 회고 때마다 우선순위에 맞게 다음 스프린트의 업무를 정함
- 매 스프린트마다 정해진 액션 아이템을 플래닝포커를 함
- 모든 구성원이 동의할 때까지 재투표
- 스프린트에 정해진 포인트만큼 업무를 가져감
- 업무를 작게 쪼개는 연습
- 리뷰와 QA 양도 적당해짐

# 좋은 제품 팀
- 지속 가능한 개발 프로세스
  - 효율적인 업무 프로세스를 위해 프로세스를 계속 최적화함
- 피드백 루프
	- 다양한 배경과 경험을 가진 팀원들의 시각을 존중하여 더 나은 제품을 만들어냄
- 고객 중심 사고
  - 개방적인 피드백을 통해 문제점을 빠르게 파악하고 개선하며, 팀의 성장을 촉진함
- 데이터 기반 의사결정
  - 팀원 간의 활발한 소통과 협력은 문제 해결 능력과 창의성을 높임
- 강한 팀워크와 문화
  - 제품에 대한 열정과 헌신은 팀원들의 동기 부여를 높이고, 최고의 결과를 만들어낼 수 있도록 이끌어줌
- 민첩성과 적응력
  - 빠른 반복과 피드백, 변화 수용
